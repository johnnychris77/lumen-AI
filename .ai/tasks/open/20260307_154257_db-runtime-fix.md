# Task: db-runtime-fix

## Owning Agent
backend

## Problem
The API and worker have inconsistent DB import/runtime behavior. Recent startup failures show Base is being imported from the wrong module, and shared runtime must not fall back silently to SQLite.

## Goal
Make API and worker use one explicit Postgres-backed DB session layer with consistent imports and no SQLite fallback.

## Scope
- app.db.base
- app.db.session
- app.db.__init__
- app.db.models
- dependent imports in API/worker startup and deps
- runtime verification in containers

## Out of Scope
- frontend changes
- auth redesign
- model behavior changes
- staging deployment

## Acceptance Criteria
- [ ] Base is imported from db.base
- [ ] SessionLocal and engine are imported from db.session
- [ ] No code imports Base from session.py
- [ ] DATABASE_URL is required
- [ ] No SQLite fallback remains in shared runtime
- [ ] API container boots successfully
- [ ] worker container boots successfully
- [ ] API and worker report the same Postgres engine URL
- [ ] inspections table exists in shared DB
- [ ] inspect flow can enqueue and process without DB import/runtime failure

## Validation Plan
- docker compose -f docker-compose.prod.yml down -v
- docker compose -f docker-compose.prod.yml up -d --build
- docker compose -f docker-compose.prod.yml ps
- docker logs --tail=120 lumen-ai-api-1
- docker logs --tail=120 lumen-ai-worker-1
- docker exec -i lumen-ai-api-1 sh -lc 'python - <<PY
from app.db import engine
print(engine.url)
PY'
- docker exec -i lumen-ai-worker-1 sh -lc 'python - <<PY
from app.db import engine
print(engine.url)
PY'
