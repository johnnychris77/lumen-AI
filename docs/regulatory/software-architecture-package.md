## 1. Architecture Overview

LumenAI is a multi-tier cloud-hosted Software as a Medical Device (SaMD) application with the following primary tiers:

```
[Client Layer]
  React PWA (Browser/Mobile)
       ↓ HTTPS (TLS 1.2+)
[API Layer]
  FastAPI REST Backend (Python 3.11+)
  JWT Authentication | RBAC | Rate Limiting | Tenant Isolation
       ↓
[Business Logic / AI Layer]
  Inspection Engine | CV Inference | Ranking Engine | Analytics
       ↓
[Data Layer]
  PostgreSQL (Production) | SQLAlchemy ORM
       ↓
[Integration Layer]
  External connectors (MedWatch, EHR, CMMS, etc.)
```

**Architecture Principles**:
1. **Human-in-the-loop**: No AI output triggers autonomous action — all findings require human confirmation
2. **Tenant isolation**: Every database query is scoped to the authenticated tenant
3. **Defense in depth**: Authentication, authorization, rate limiting, and audit logging at every tier
4. **Fail safe**: When AI is unavailable, manual inspection workflows remain accessible
5. **Explainability**: AI confidence scores and ranking rationale exposed to human reviewer

## 4. Security Boundaries & Trust Zones

### Zone 1 — Public (Unauthenticated)
- **Endpoints**: /health, /status, /api/integrations/catalog
- **Data access**: None (no PHI, no tenant data)
- **Rate limiting**: Applied (DoS protection)

### Zone 2 — Authenticated (Standard Access)
- **Endpoints**: All /api/ routes not in Zone 1, 3, or 4
- **Authentication**: JWT bearer token required
- **Authorization**: Tenant scope enforced on all queries
- **Data access**: Own tenant data only
- **Rate limiting**: Applied per user/tenant

### Zone 3 — Enterprise (Feature-Gated)
- **Endpoints**: /api/executive/*, /api/network/*, /api/patient_safety/*, /api/copilot/*
- **Authentication**: JWT + enterprise tier verification (tier_guard middleware)
- **Authorization**: require_enterprise_auth decorator applied
- **Data access**: Own tenant data + anonymized network data (no cross-tenant PHI)
- **Rate limiting**: Applied with higher limits for enterprise tier

### Zone 4 — Admin (Elevated Privilege)
- **Endpoints**: /api/admin/*, /api/audit/*
- **Authentication**: JWT + admin role assertion
- **Authorization**: Admin role required (RBAC check)
- **Data access**: Cross-tenant administrative functions (audit logs, tenant management)
- **Rate limiting**: Strict limits applied; all access logged

## 5. Tenant Isolation Architecture

Multi-tenant isolation is enforced at multiple layers:

**Layer 1 — Authentication**: JWT token contains tenant_id claim. All requests are authenticated and tenant_id is extracted from the verified token.

**Layer 2 — Database Query Filtering**: Every SQLAlchemy query that accesses tenant data applies `.filter(Model.tenant_id == current_tenant_id)`. Cross-tenant joins are not implemented; there are no queries that return data across multiple tenants.

**Layer 3 — Route-Level Verification**: Route handlers extract tenant_id from JWT and pass it explicitly to service functions. Service functions accept tenant_id as a required parameter.

**Layer 4 — Network Intelligence Pseudonymization**: When facility data is aggregated for network benchmarking (Module F), facility identifiers are replaced with pseudonymous IDs using a salted hash. Salt is rotated monthly. Aggregations with fewer than 5 facilities are suppressed (k-anonymity, k≥5).

**Layer 5 — Audit Logging**: All cross-tenant admin access is logged with admin user_id, target tenant_id, action, timestamp, and IP address.

## 6. Mobile Architecture

**Progressive Web App (PWA) Design**:
- Frontend is a React application built with Vite, deployed as a PWA
- Service Worker registered at application startup manages offline caching
- IndexedDB provides structured offline storage for inspection sessions

**Offline Inspection Flow**:
1. User creates inspection session (online or offline)
2. If offline: session stored in IndexedDB with state = PENDING_SYNC
3. PENDING_SYNC badge displayed prominently to user
4. When connectivity restored: Service Worker triggers sync
5. Sync endpoint: POST /api/mobile/sessions/{id}/sync
6. On success: session state updated to SYNCED; badge removed
7. On failure: retry with exponential backoff (max 5 attempts)
8. On max retries exceeded: SYNC_FAILED alert generated; user notified

**Mobile Image Capture**:
- Browser camera API (getUserMedia) for image capture
- Minimum compression quality floor: ≥ 0.85 (JPEG quality) to preserve defect detail
- Images stored in IndexedDB as base64 until sync
- Image upload via multipart form to /api/inspection/upload on sync

## 7. AI Inference Architecture

**Inference Pipeline**:
1. Image received at /api/inspection/submit endpoint
2. Image preprocessed (resize, normalize) by inference adapter
3. CV model inference executed (target: < 10 seconds p95)
4. Model returns: finding_type, confidence_score (0.0–1.0), bounding_box, model_version
5. Results stored in inspection_findings table with all metadata
6. Results returned to frontend for human reviewer display
7. Human reviewer views findings, image evidence, confidence score
8. Human confirms or overrides each finding (action logged)
9. Final disposition recorded in inspection_record table

**Model Version Tracking**:
- Each finding record stores model_version (e.g., "MODEL-A-1.2")
- Model changes follow AI/ML Change Control Plan (docs/regulatory/ai-ml-change-control-plan.md)
- Significant changes trigger re-validation per Predetermined Change Control Plan

**Human Confirmation Gate**:
- No automated disposition is written to the quality record without human action
- `human_reviewed = False` is the default state for all AI findings
- Pass/fail disposition record requires `human_reviewed = True` and `reviewer_id` populated
- This gate is enforced at the data layer — not just the UI layer

## 8. Integration Architecture

**Connector Pattern**:
- All external system integrations implemented via `BaseConnector` abstract base class
- Each connector implements: `connect()`, `pull_events()`, `normalize_event()`, `disconnect()`
- Events are normalized to standard internal models before storage
- PHI is stripped at the normalization layer — de_identified=True enforced

**Implemented Connectors** (12):

| Connector | External System | Data Type | PHI Risk | Control |
|-----------|----------------|-----------|---------|---------|
| MedWatch | FDA Recall Database | Device recalls | None | N/A |

## 9. Data Flow Description

### 9.1 Standard Inspection Flow
```
User (Browser) → HTTPS → Nginx (TLS termination)
  → FastAPI (/api/inspection/submit)
  → JWT validation → tenant_id extraction
  → Image stored → CV Engine invoked
  → findings returned with confidence scores
  → findings stored (inspection_findings table, tenant_id scoped)
  → response returned to browser
  → Human reviewer sees findings + image + confidence
  → Human confirms/overrides → disposition stored (human_reviewed=True)
  → Audit log entry written
```

### 9.2 Offline Sync Flow
```
Mobile browser → IndexedDB session (PENDING_SYNC)
  → Network restored → Service Worker sync trigger
  → POST /api/mobile/sessions/{id}/sync
  → JWT validated → tenant_id extracted
  → Session data written to PostgreSQL
  → CV inference executed on uploaded images
  → Session state = SYNCED
  → Audit log entry written
```

### 9.3 Network Intelligence Flow
```
Facility data (tenant_id scoped)
  → Pseudonymization (salted hash of facility_id)
  → Aggregation (minimum k=5 facilities)
  → Network benchmark table (no tenant_id, only cohort_id)
  → Benchmark API returns cohort statistics only
  → No individual facility data exposed
```

## 10. Cryptography Specification

| Usage | Algorithm | Notes |
|-------|-----------|-------|
| Transport encryption | TLS 1.2+ | Required in production; enforced by Nginx |
| Password hashing | bcrypt (cost factor ≥ 12) | Python bcrypt library |
| JWT signing | HS256 (current); RS256 (production target) | Asymmetric signing planned |
| JWT secret | Minimum 32-character random string | Stored in environment variable |
| Pseudonymization | SHA-256 with rotating salt | Monthly salt rotation; k-anonymity k≥5 |
| Database encryption at rest | Cloud provider managed (AES-256) | AWS RDS encryption |
| Secrets storage | Environment variables / AWS Secrets Manager | Never hardcoded |

**Cryptographic Gaps Identified**:
- JWT signing is currently HS256 (symmetric) — RS256 (asymmetric) planned for production
- Database field-level encryption not implemented (rely on disk encryption at rest)
- End-to-end image encryption not implemented (TLS in transit + disk encryption at rest)
