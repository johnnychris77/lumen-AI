Objective 1 review. Tracks real, verified defects found during this maintenance phase's regression investigation — not a hypothetical template. Every entry is grounded in a reproduced failure, a root-cause trace, and (where fixed) a commit reference.

| BUG-001 | Medium (AI reasoning defect) | `app/services/sentinel_dashboard_service.py` — `_compute_enterprise_risk_score` | `quality_risk = 100 - quality_score` was never clamped to [0,100] like the other three risk factors (`watchlist_pressure`, `alert_pressure` both use `min(100, ...)`). `quality_score` (from `quality_dashboard_service.executive_quality_score`) can go negative when `repeat_rate` (repeated-error events ÷ supervisor-correction events) exceeds 100% — plausible whenever repeat-error events outnumber correction events. This pushed `enterprise_risk_score` above its documented 0-100 bound (observed: 105). | Engineering | Clamped `quality_risk` with `max(0, min(100, 100 - quality_score))`, matching the pattern already used for the sibling factors. Commit `c065c20`. | **Fixed and verified** — `tests/test_sentinel_orchestration.py::TestDashboardAggregation::test_dashboard_returns_expected_shape` passes after the fix. | 1.0.1 |
| BUG-002 | Low (test-infrastructure defect, not a product defect) | `backend/test.db` (local dev/test SQLite file) | The gitignored, file-based test database is never reset between pytest invocations. Tests using hardcoded, non-unique identifiers (e.g. `technician="tech-apollo-1"`, `zone="unique_zone_xyz"`) accumulate matching rows across every repeated local run, causing exact-count assertions (`assert result["annual_competencies"] == 1`) to fail once enough historical runs have polluted the table. This produced 17-19 apparent "failures" across `test_apollo_quality.py`, `test_federated_horizon.py`, `test_genesis_ai_intelligence_cloud.py`, `test_guardianx_assurance.py`, `test_industry_collaboration.py`, `test_pilot_deployment.py`, `test_quality_guardian.py`, and `test_quality_intelligence.py`. **Verified this never affects CI**: `.gitignore` already excludes `*.db`, and every CI workflow runs against a fresh runner with no pre-existing `test.db`. | Engineering | Deleted the stale local `test.db`; a single fresh full-suite run passed 3381/3381 (2 skipped, 0 failed) — confirming this is 100% a local dev-environment artifact, not a product regression. No code fix needed or applied. | **Resolved by environment reset** — not a code regression | N/A (local environment only) |
| BUG-003 | Low (test-reliability defect, not a product defect) | `backend/tests/test_sentinel_orchestration.py` | `TestRiskMonitor`, `TestDigitalTwinMonitoring`, and `TestAlertGeneration` all share one fixed module-level `TENANT` constant with no per-test unique-identifier helper (unlike ~15+ other test files in this suite, which use a `uid(prefix)` helper to generate unique tenant IDs per test). This makes exact-count/absence assertions in this file order-dependent on which other tests in the same class ran first within a session. | Engineering | Documented, not fixed — resolving this properly requires adopting the `uid()` pattern already used elsewhere in the suite, which touches many tests in this file and was judged too broad for a "small, safe" patch-level fix in this cycle. | **Open — recommended for Phase 8 test-hardening backlog** | Targeted for 1.0.5 (reliability improvements) |

## Bug categories represented (per Objective 1's examples)

- **AI reasoning defects**: BUG-001 (composite score exceeding its documented bound).
- **Data defects**: BUG-002 (test-fixture data accumulation — local-only, not a product data-integrity issue; see `docs/production-readiness/PRODUCTION_READINESS_SCORECARD.md`'s TD-02/17/18 for the separate, real production database-integrity gaps, which are unrelated to this finding).
- **Performance defects**: see `docs/release-management/PERFORMANCE_LOG.md` for the N+1 query pattern in `atlas_dashboard_service.py` and the inconsistently-queued AI inference route — both are real, open items tracked there rather than duplicated in this register, since neither has yet been fixed.

## What did not make this register

Findings from prior phases (Docker `$PORT` default, `docker/Dockerfile.worker` stub, empty `ci.yml` step) are deployment/CI configuration issues, not application bugs — they remain tracked in `docs/commercial-readiness/DEPLOYMENT_GUIDE.md` and `docs/commercial-readiness/PRODUCT_OPERATIONS_GUIDE.md` rather than duplicated here, per this program's scope boundary between "bug management" (application defects) and "deployment/product operations" (infrastructure and process).
