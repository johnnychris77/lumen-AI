# LumenAI Multi-Site Rollout Playbook

> **Scope:** This playbook guides enterprise deployment teams through scaling LumenAI from a single-site pilot to a multi-hospital health system deployment.

---

## 1. Pre-Conditions for Expansion

Before initiating multi-site rollout, the originating pilot site must meet all of the following:

| Criterion | Minimum Threshold |
|-----------|-------------------|
| Facility Readiness Score | ≥ 80 (Ready) |
| Inspection Volume (30-day) | ≥ 50 inspections |
| Contamination Detection Accuracy | ≥ 90% (human-confirmed) |
| Training Completion | 100% of required personnel |
| CAPA Resolution Rate | ≥ 75% of open items closed |
| Pilot Duration | ≥ 90 days of active use |

> **Note:** All expansion recommendations generated by LumenAI analytics are candidate signals only. Final go/no-go decisions require human review by qualified clinical and operational leadership.

---

## 2. Enterprise Hierarchy Structure

LumenAI uses a five-tier hierarchy:

```
HealthSystem
  └── Market (geographic or business unit grouping)
        └── Region (sub-market operational cluster)
              └── Facility (individual hospital or ASC)
                    └── Department (SPD, OR, ICU, etc.)
```

Each tier is provisioned via the `/api/enterprise` endpoints and must exist before subordinate tiers can be created.

---

## 3. Rollout Phases

### Phase 1 — Health System Registration

1. Create the `HealthSystem` record via `POST /api/enterprise/health-systems`
2. Assign a `system_id` (lowercase alphanumeric, 4–64 chars, e.g., `midwest-health`)
3. Record the `admin_email` for system-level notifications
4. Confirm `contract_tier` (enterprise | standard | pilot)

### Phase 2 — Market & Region Setup

1. Create Markets under the Health System: `POST /api/enterprise/markets`
2. Create Regions under each Market: `POST /api/enterprise/regions`
3. Verify hierarchy lookup at `GET /api/enterprise/hierarchy/{system_id}`

### Phase 3 — Facility Onboarding

For each facility:

1. Create Facility record: `POST /api/enterprise/facilities`
2. Start onboarding workflow: `POST /api/enterprise/onboarding/start`
   - `workflow_type: "site"`
3. Progress through workflow steps in order:
   - `initiated` → `documents_collected` → `tenant_provisioned` → `users_invited` → `baseline_assigned` → `training_complete` → `go_live` → `completed`
4. Advance steps via: `POST /api/enterprise/onboarding/{workflow_id}/advance`
5. Confirm `GET /api/enterprise/onboarding/{workflow_id}` shows `completed`

### Phase 4 — Baseline Distribution

1. Upload instrument baselines for each facility
2. Submit for approval: `POST /api/enterprise/baselines`
3. Approve via authorized reviewer: `POST /api/enterprise/baselines/{id}/approve`
4. Publish to facilities: `POST /api/enterprise/baselines/{id}/publish`
   - Only baselines with `approval_status: "approved"` can be published
   - `published_to` stores the list of facility IDs receiving the baseline

### Phase 5 — Department Setup

1. Create departments under each facility: `POST /api/enterprise/departments`
2. Assign `department_type` (spd | or | icu | cath_lab | endoscopy | other)
3. Assign `manager_email` for department-level communications

### Phase 6 — Go-Live Validation

Before activating each site:

- Confirm readiness score ≥ 80 via `GET /api/enterprise/dashboards/readiness`
- Review system quality dashboard: `GET /api/enterprise/dashboards/system-quality`
- Confirm no outlier facilities with contamination rate > 2× system average
- Executive scorecard review: `GET /api/enterprise/dashboards/executive-scorecard`

## 4. Onboarding Workflow Reference

### Site Onboarding Steps (in order)

| Step | Description | Responsible Party |
|------|-------------|-------------------|
| `initiated` | Facility identified, kickoff scheduled | Enterprise Deployment Team |
| `documents_collected` | Contracts, facility data forms received | Contracts / Legal |
| `tenant_provisioned` | LumenAI tenant created, credentials issued | Platform Team |
| `users_invited` | User accounts created and invited | IT / Admin |
| `baseline_assigned` | Instrument baselines assigned and approved | Quality / SPD Leadership |
| `training_complete` | All required personnel trained and certified | Training Team |
| `go_live` | System activated for production use | Deployment Team |
| `completed` | Monitoring phase confirmed, handoff complete | Customer Success |

### User Onboarding Steps

`initiated` → `identity_verified` → `role_assigned` → `training_complete` → `completed`

### Vendor Onboarding Steps

`initiated` → `agreement_signed` → `baseline_submitted` → `baseline_reviewed` → `approved` → `completed`

### Baseline Onboarding Steps

`initiated` → `uploaded` → `qa_review` → `approved` → `published` → `completed`

## 5. Parallel Site Rollout Strategy

For health systems onboarding 3+ facilities simultaneously:

1. **Wave approach:** Group facilities into waves of 2–3 sites maximum
2. **Stagger go-live dates** by at least 2 weeks per wave
3. **Designate a lead facility** in each wave to serve as reference site
4. Monitor system-level adoption weekly via `/api/enterprise/dashboards/adoption`
5. Do not advance Wave 2 until Wave 1 facilities reach readiness score ≥ 70

---

## 6. Risk Indicators During Rollout

The following signals warrant deployment pause and human review:

- Facility readiness score drops below 60 after go-live
- Contamination rate at new site is > 2× the system average at 30 days
- Training completion stalls below 80% at 2 weeks post-kickoff
- Baseline coverage below 70% at go-live
- CAPA backlog growing week-over-week for 3+ consecutive weeks

> All risk signals are candidate indicators only. Investigation and disposition require qualified human review.

---

## 7. Data Isolation Guarantee

Each facility operates under its own `tenant_id`. LumenAI enforces strict tenant isolation:

- All database queries include `tenant_id` filters
- Analytics dashboards only surface aggregate, anonymized cross-site data
- No facility can access another facility's raw inspection data
- Cross-site benchmarking uses anonymized identifiers only
- Every cross-site data access event is logged to the audit trail

---

## 8. Rollback Procedure

If a site deployment must be paused or reversed:

1. Set facility `is_active: false` via admin panel
2. Onboarding workflow can be queried but cannot be reversed (audit record is immutable)
3. Contact enterprise support to deprovision tenant credentials
4. Retain all inspection and audit data per your organization's retention policy
