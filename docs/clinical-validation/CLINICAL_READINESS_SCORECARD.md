# LumenAI — Clinical Readiness Scorecard

**Clinical Validation Program · Phase 3: Florence · Clinical Review, Scientific Validation & Patient Safety Assurance**

Objective 15 roll-up of every review dimension in this Phase 3 audit. Ratings are **Strong / Adequate / Needs Work / Critical Gap**, each tied to a specific finding elsewhere in this document set — no rating here is unsupported or invented. This scorecard is the clinical companion to Phase 1's [`PRODUCTION_READINESS_SCORECARD.md`](../production-readiness/PRODUCTION_READINESS_SCORECARD.md); it does not repeat that document's architecture/infrastructure findings except where they directly bear on clinical safety.

| Dimension | Rating | Basis |
|---|---|---|
| Clinical accuracy | **Needs Work** | No trained model weights ship in this repository — the deployed `_deterministic_fallback()` emits only `debris`/`corrosion` plus non-finding placeholders `stain`/`clean`, a small fraction of the 12-13-category design taxonomy. Pre-market sensitivity/specificity/kappa figures are targets from mock/synthetic data; the planned multi-site blinded reader study has not been conducted — [CLINICAL_SCOPE.md](./CLINICAL_SCOPE.md), [FINDING_TAXONOMY.md](./FINDING_TAXONOMY.md) |

| Evidence quality | Strong | Veritas's evidence-readiness score is built from real, itemized penalties (baseline-match quality, governance status, image quality, anatomy coverage, identity confidence, conflicts) with a genuine insufficiency gate (`GATE_ANALYSIS_BLOCKED` below score 50) rather than an unexplained number or silent low-confidence pass-through — [CLINICAL_RECOMMENDATIONS.md §Evidence requirement](./CLINICAL_RECOMMENDATIONS.md) |

| Knowledge quality | Needs Work | `ifu_reference` (6 model files), `KnowledgeArticle.references`, and Oracle's `supporting_literature_json` are universally unstructured free text/JSON with no citation-validation logic anywhere in the codebase — no format, existence, or validity check was found for any manufacturer or clinical-literature reference field — [AI_LIMITATIONS.md §Knowledge governance has no review-schedule enforcement](./AI_LIMITATIONS.md) |
| Digital twin maturity | Adequate | `digital_quality_twin.py`'s forecast/simulation/intervention/executive-brief models consistently pair every projection with `confidence_score`, `human_review_required`, and an explicit non-causation disclaimer, plus a genuine prediction-vs-actual calibration mechanism (`ForecastOutcome`). Apollo's governance-health twin, by contrast, has no numeric confidence field at all, and its `twin_history` view drops even the qualitative scope-limitation disclosure present in the live-compute path — the clearest, lowest-risk improvement this review identifies — [DIGITAL_TWIN_CLINICAL_MODEL.md](./DIGITAL_TWIN_CLINICAL_MODEL.md) |

## Overall assessment

**LumenAI's patient-safety architecture is genuinely strong where it matters most**: no reviewed code path allows an AI output to become an irreversible action without a tier-checked human role, every consequential override requires a written reason, and the codebase consistently prefers an explicit "insufficient/blocked" state over a silently-accepted low-confidence guess. This discipline is structural — enforced in code, not just asserted in documentation — and is the single strongest finding of this Phase 3 review, mirroring Phase 1's finding about the platform's architectural coherence.

**One finding should be surfaced above all others in any clinical-facing description of the platform**: no trained model ships in this repository today. The deployed inference path produces only `debris`/`corrosion` — a small fraction of the 12-13-category taxonomy the scoring and education layers are designed around. Every clinical-accuracy claim elsewhere in this document set (or in prior marketing/pitch material) must be scoped to that reality.

**Three Needs-Work items reflect a consistent pattern — a well-designed conceptual model that is not fully reflected in the enforced system** — and should be the opening backlog for the next phase, in priority order:
1. **Human oversight role-mapping gap** — clinical stakeholders reading "Supervisor," "Director," or "Manufacturer" in this platform's documentation will reasonably expect a distinct, independently-enforced role; only two of five conceptual tiers actually have one. This should be corrected either by adding the missing roles or by renaming the conceptual tiers to match what is actually enforced.
2. **Workflow-stage coverage gap** — only half of the classical 8-stage SPD cycle is tracked, split across two un-unified models. Unifying "cleaning," "assembly," and "pre-sterilization review" into real, timestamped stages (rather than RCA labels or computed reports) would close the largest remaining gap between the platform's workflow narrative and what it actually observes.
3. **Knowledge governance and reference-quality gaps** — no review-schedule enforcement and no citation validation on any manufacturer/literature reference field. Low implementation cost relative to safety value, since the governance scaffolding (approval workflow, versioning) already exists and only needs two additional fields and a validation pass.

**Explainability and digital twin maturity are Adequate, not Strong, for the same underlying reason**: the best pattern in the codebase (Veritas's field completeness; `digital_quality_twin.py`'s consistent hedging) is real and should be replicated, but is not yet uniform across every AI-output table or every twin system (Sentinel-X/Oracle lack an explicit limitations field; Apollo's twin lacks a confidence field). This is a consistency problem, not a missing-capability problem, and is the cheapest category of improvement available.
