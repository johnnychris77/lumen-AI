# LumenAI — Clinical Validation Plan

Objective 13 review. This document does not duplicate the pre-existing, more detailed regulatory validation planning already in `docs/clinical/clinical-validation-plan.md` (LUM-CVP-001 v1.0) and `docs/clinical/clinical-safety-review.md` — it references them and adds the one thing this Phase 3 review was specifically asked for: a representative clinical scenario suite with documented expected AI behavior, cross-checked against what's actually tested today.

## What the existing plans already cover (reference, don't restate)

- **`docs/clinical/clinical-validation-plan.md`**: FDA Class II/510(k) framing, ISO 14971, IEC 62304 Class B, AAMI TIR45; validation objectives (Cohen's kappa ≥0.80, critical-finding sensitivity ≥95%, specificity ≥80%, critical false-negative rate ≤2%); a 3-site blinded multi-site reader study design with stratified sampling and a 3-person consensus ground-truth panel; a governance body (CVC); change-control/revalidation triggers; a 12-month timeline.
- **`docs/clinical/clinical-safety-review.md`**: ISO 14971 hazard analysis (6 hazards, all currently rated acceptable/RPN<15); the escalation workflow (AI flags → supervisor notified → accept/override with mandatory co-sign on CRITICAL findings); override-rate KPI thresholds (>20%/>40% trigger review); a safety-controls table (confidence threshold 0.70, drift alerts, model version pinning, audit integrity); post-market surveillance cadence.
- **Real, queryable backing tables already exist** (`app/models/validation.py`, P12) for exactly this kind of validation work: `ValidationCase` (ground truth vs. AI/human prediction per case), `ValidationRun` (aggregated precision/recall/specificity/F1/kappa/AUC per run), `SealedTestRegistry` (hash-sealed test sets with a pass/fail gate), `RWEEnrollment`/`RWEMetricSnapshot` (real-world-evidence facility enrollment and weekly metrics), and `ShadowPrediction` (silent-mode predictions reconciled against the human's actual decision, never shown clinically). These are not aspirational — they are real ORM tables that the metrics in the documents above are computed from.
- **`docs/ai/safety-metrics.md`, `docs/ai/model-evaluation-framework.md`, `docs/ai/shadow-mode-validation.md`** define the actual per-KPI metric computation (precision/recall/F1/FPR/FNR), reviewer-agreement/kappa calculation, and the shadow-mode → review → staged-enable → monitor promotion process, including a `worst_safety_false_negative_rate` rollup specifically tracked for blood/tissue/organic-residue/crack/missing-component.

**Important honesty note carried forward from [CLINICAL_SCOPE.md](./CLINICAL_SCOPE.md)**: the sensitivity/specificity/kappa figures in these existing documents are pre-market **targets**, backed by **mock/synthetic data** — the planned multi-site blinded reader study has not yet been conducted. This plan does not change that status; it should not be read as evidence the study is complete.

## Representative clinical validation scenarios (Objective 13)

Each scenario below states expected AI behavior and whether a real automated test already exercises it.

| # | Scenario | Expected AI behavior | Test coverage |
|---|---|---|---|
| 1 | Retained blood | Detected as `blood` finding type (part of the stable-core taxonomy present in every internal vocabulary); severity scored via the probability→severity_index mapping; high-severity blood findings feed the safety-critical FN-rate tracking in `safety-metrics.md` | **Covered** — `test_ranking_engine.py`, `test_cv_pipeline.py` |
| 2 | Superficial staining | Distinguished from deeper contamination via the `_SEVERITY_SCALES` label progression (none/trace/visible/heavy for blood-family findings); risk-weighted lower than active retained soil in Sentinel-X's SPD risk matrix (`RISK_WEIGHT_LOW`) | **Partial** — no test named for this exact scenario; the underlying staining/discoloration finding type is exercised generically in `test_vulcan_reliability.py::test_stable_cosmetic_discoloration_does_not_trigger_removal`, `test_sentinel_x_risk.py`, `test_spd_risk_weighting.py`. **Recommend adding a scenario-named test** as a Phase 4 follow-up |
| 3 | Progressive corrosion | Vulcan's progression model classifies the trend (`rapidly_worsening`/`slowly_worsening` per the exact severity-diff rules in [DIGITAL_TWIN_CLINICAL_MODEL.md](./DIGITAL_TWIN_CLINICAL_MODEL.md)), never asserts a trend from a single observation | **Covered** — `test_vulcan_reliability.py::test_repeated_corrosion_creates_progression_signal` |
| 4 | Damaged O-ring | Classified via `instrument_anatomy.py`'s lumen/scope zone family; flagged at Sentinel-X's `RISK_WEIGHT_HIGHEST` tier given its association with lumen integrity | **Covered** — `test_vulcan_reliability.py::test_rigid_scope_o_ring_failure_uses_correct_anatomy_language` |
| 5 | Cracked drill flute | Classified via the `rotary_orthopedic` zone category; anatomy-aware finding-type language confirmed (not a generic "crack" label divorced from instrument context) | **Covered** — `test_vulcan_reliability.py::test_drill_bit_flute_failure_uses_correct_anatomy_language`, plus anatomy-specific tests in `test_instrument_zones.py`, `test_v2_0_anatomy_aware_clinical_intelligence.py`, `test_instrument_architecture_preservation.py` |
| 6 | Incomplete anatomy coverage | `inspection_coverage.compute_coverage` flags the gap; Veritas's evidence-readiness score is penalized (limited/insufficient coverage subtracts 15-30 points), lowering `readiness_category` and potentially triggering `GATE_ANALYSIS_BLOCKED` | **Covered** — `test_veritas_evidence.py::test_missing_critical_zone_lowers_evidence_readiness` |
| 7 | Poor image quality | Image-quality penalty applied in Veritas's readiness scoring; explicit recapture guidance surfaced rather than a low-confidence conclusion being silently accepted | **Covered** — `test_veritas_evidence.py::test_poor_image_quality_requires_recapture_guidance` |
| 8 | Baseline unavailable | An unapproved or superseded baseline is never used to drive final scoring (`baseline_comparison_scoring_service.resolve_baseline`'s governance check); readiness score penalized for missing/uncertain baseline match | **Covered** — `test_veritas_evidence.py::test_unapproved_baseline_cannot_drive_final_scoring`, `test_superseded_baseline_is_not_selected`; broader coverage in `test_baseline_ranking_contract.py`, `test_clinical_readiness.py`, `test_p26_baseline_governance.py` |
| 9 | Conflicting evidence | A `VeritasEvidenceConflict` row is created (e.g. anatomy-zone mismatch between two evidence sources) rather than silently picking one side; conflict resolution requires a named responsible reviewer role | **Covered** — `test_veritas_evidence.py::test_anatomy_zone_mismatch_creates_evidence_conflict` |

**Bonus scenario found during this review, not in the brief's example list but directly relevant**: insufficient history for a progression claim (fewer than 2 prior findings) — **covered** by `test_vulcan_reliability.py::test_insufficient_history_produces_limited_confidence_result` and `test_sentinel_orchestration.py::TestDigitalTwinMonitoring::test_insufficient_history_not_flagged`.

**Net result: 8 of 9 brief-named scenarios have direct, real automated test coverage today.** Only "superficial staining" lacks a test scoped to that exact scenario name, though the underlying finding type is exercised by more general tests. This is a strong result and should be reported as such — it reflects real, existing test discipline, not newly-written tests created for this review.

## What this document adds vs. what already existed

This is the first place in the documentation set that explicitly cross-references the brief's 9 example clinical scenarios against real test names — prior validation documents describe *statistical* validation methodology (kappa, sensitivity, reader-study design) but did not previously enumerate scenario-level behavioral expectations tied to specific tests. Recommend this table be kept current as new scenario tests are added.
